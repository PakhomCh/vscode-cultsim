The Inner Workings of Cultist Simulator (How it works + Modding guide)

Introduction

Preface

If you’re reading this guide, it probably means that you’re interested in modding Cultist Simulator. This document was compiled by several modders in the community who felt that the resources available to learn modding were, at the time of writing, outdated and often inadequate. Although there is good mod support for Cultist Simulator, the process of writing mods can be tricky and often requires a lot of guesswork and problem solving. This makes the barrier for entry to modding rather high, especially for people without a background in computer science.

This guide is meant to provide up-to-date information to help people make mods with as few difficulties as possible, as well as understanding better “how to do” things, or how things already work inside the game. We hope that it can act as both a learning tool and a reservoir of knowledge for aspiring modders.

Building Blocks of Cultist Simulator

Content Format

The content of Cultist Simulator is written in JSON files. JSON files are text files following the JSON standard.

Each file can only contain one type of content (element, recipe, deck, verb, ...). Not doing this will trigger a crash on startup.

Each files follows the exact same structure:

For each content file, there is a top level object containing one or more lists/arrays of objects. Only one list/array of one type can be included per file; but you can add as many files as you like. These lists lists/arrays are named one of the following: "elements", "recipes", "decks", "legacies", "endings", or "verbs".

Modifying Base-Game Resources

Merge-Overwriting

If you create a recipe, element, legacy, ect. and set the ID to the same as a base-game (or another mod’s) recipe, element, legacy, ect. they will merge-overwrite. This is a behaviour where you can apply changes to the base-game resource without having to redefine everything about it, allowing other mods to change the remaining parts.

Internal decks have deck IDs

Internal decks have externally-defined deck IDs. These IDs are defined as “deck.” followed by the recipe ID.

E.G. to modify the internal deck belonging to the recipe “explore.fleeting”, you simply use the deck ID “deck.explore.fleeting”.

This will not make any change to the card’s label, icon, description, uniqueness, or slots.

Overwriting

Aside from the “big six”, objects in Cultist Simulator can’t be merge-overwritten, and can only be overwritten. This is as simple as redefining the same object using the same ID.

  • For images, if an image with the same name is put in a mod folder, it will automatically be used instead of the base-game asset with the same name.
  • Properties of the six/seven main object types cannot be merge-overwritten. This means that slots, numbers, strings, booleans, and other properties cannot be modified without being overwritten... though certain allowances are made through property operations.
  • Localization files use this same principle when overwriting text from the base game. Due to an oversight in the Fucine modloader, only the last-loaded localization change will apply to a given string. In general, this means that non-Chinese localization can only work if no Chinese localization is installed, regardless of the game’s (non-English) language setting.

Property Operations

When merge-overwriting, it is possible to adjust some properties without replacing them: namely lists and dictionaries. These adjustments are made through property operations. Using a property operation is done by adding a $ and the operation name to the name of the property to be modified.

Lists are ordered, and thus support $prepend and $append, while dictionaries are unordered and thus support $add instead; both lists and dictionaries support $remove. (More esoteric, unsupported property operations exist, such as $plus and $minus for integers.) These are collectively referred to as “property operations”.

Note: even if “aspects” is a dictionary (it uses {curly braces} and contains string:number mappings), “aspects$remove” is a list (it uses [square brackets] and contains only strings). This is because the removal of the aspect from the list is not dependent on the aspect’s level: only the aspect itself. The same holds true for XTriggers and their catalysts, as well as any other dictionaries in the game files and their keys.

Note: even though a deck’s contents are shuffled, the deck specification is ordered. As a result, the deck spec supports $prepend and $append, and not $add.

Limitations of Property Operations:

  • One major restriction of property operations is that they only work if the list/dictionary being adjusted already exists on the resource. For example, if an element has no aspects, you cannot use "aspects$add" on that element. This means that the aspect list must be overwritten instead.
  • Property operations cannot be applied to properties defined in other mods, even if those mods were loaded first.
  • While lists and dictionaries constitute most of the complex data found in the Cultist Simulator files, there are no property operations for Slots. The first and foremost purpose of AchieveIntegrations is to overwrite slots in ways that other mods can use, promoting compatibility: AchieveIntegrations overwrites Slots to support new custom aspects that allow cards to be used, or forbid cards from being used, in base-game Slots without other mods having to overwrite the Slots.
  • Due to a bug in the game engine, $remove does not work on lists of Slots.
  • Due to an oversight in the Fucine modloader, a given property operation can only be applied to a given property once, across all installed mods. If more than one mod attempts to use the same property operation on the same property, only the mod that is loaded last will have its property operation applied, similar to overwriting. (There is currently a DLL patch that fixes this issue, which is included as part of GreatWork. GreatWork is an unlisted Steam mod that needs the player to install BepInEx before it can work.) Despite this, it is still recommended that modders use property operations instead of replacement when possible.
  • Attempting to use a property operation on something that is not the right data type (i.e. using $append on a dictionary or $add on a list) will crash the game on startup.

Elements

Description

Elements are the nouns of the sentence. They are things that are.

Elements encompass two categories; cards and aspects. Cards are the nouns, the things you use in the game. They are, among other things, defined by a set of aspects. The aspects are special elements with fewer properties, that can be seen as metadata, or a more granular type of information we can act on.

Red = Card. Green = Aspects of the “London” card.

The cards on the table are elements. The aspects of cards are also elements. Everything that’s represented as a card, from Lore, Books, Patrons, Influences, is an element.

Properties of all Elements

ID ("id")

The unique identification of the element. This is used to refer to the element. If an element with this ID already exists, the properties specified will be merged with that of the existing element. This is used to modify pre-existing elements, like those from the base game.

Spelling Causes a Crash on Startup

Any spelling mistakes made when referencing this ID will lead to an error that black screens the game at launch (see Running and Debugging Your Mod).

Spelling mistakes made when referencing this ID can create issues in your mod, such as XTriggers failing to operate, incorrect recipes being executed, or the game crashing (though not necessarily at startup.) The player log is likely to include a line that says something akin to “unknown element id specified: scholardeepmandaic”. This will indicate what element ID doesn’t exist, but will not indicate what the correct spelling is. (In this case, “scholarmandaic”.)

Label ("label")

The name displayed on the card. For aspects, the label is displayed when it's icon is clicked on, and it is prefixed with “Aspect: ”.

Description ("description")

Shown in the top-right when the aspect/card is clicked on.

Aspect ("isAspect")

For cards (“full elements”, “non aspects”), this should always be false. It defaults to false, and can be safely omitted. For aspects, this should be true. This is used by many modding tools to differentiate aspects from other elements (as the two can have very different properties), but the game only considers this value when determining where to look for the element’s art. (If the element is on a card as an aspect, the game will check “images\aspects” only if this value is set to true. If the element is a card, or if it does not have this value as true, the game will check “images\elements” instead.)

Icon ("icon")

The name of the file for the icon of this element. Cards get their art from images/elements. Aspects get their art from images/aspects (see the isAspect property for their distinction). This value defaults to the ID of the element, and can be removed if the desired image has a filename matching the ID of the element, and is in the appropriate folder.

When Not to use Special Characters

If the element id uses non-letter characters (“something.otherthing.anotherthing” or "something_otherthing"), and the icon has the same name, then it cannot be used as a status bar element. The status bar icons cannot have special characters in them, and because the name of the icon has to have the same name as the value of the “icon” field, you cannot use special characters in the value of this field if you plan on displaying this element into the status bar.

Induces ("induces")

Whenever this element is present and face-up (ie: not from effects or deckeffects) at the conclusion of a recipe chain, the induced recipe may be created as a new situation token depending on the designated chance. This will not occur if an active token with the actionId of the specified verb is already on the table, but it will ignore any requirements, extantreqs, or tablereqs associated with the induced recipe.

Examples of inductions include the Heart aspect starting 'Course of the Heart', or the Ingredient aspect starting 'needs'.

Decay Element ("decayTo")

Specific to non-aspect elements (sort of). The element that this element will turn into when the lifetime reaches 0. Mutated aspects are kept through decay. (As an advanced feature, this can be applied to aspects to achieve a specific behaviour when used with Purge, but has no effect on it's own because aspects cannot decay.)

Examples of decay elements being used include Restlessness decaying to Dread, or Health [Fatigued] decaying into Health.

Verb Icon ("verbicon")

When this element is in a Verb, the Verb’s icon will become the icon indicated. Note that the icon must exist in “images\verbs”. It is currently unknown what happens when the image does not exist or more than one Verb Icon is defined.

Properties specific to Aspects

Hidden (“isHidden”)

This causes the aspect not to appear on any in-game aspect lists.

An example of a hidden aspect is ‘legacyplotitem’, which does not appear on the aspect list of A Bequest, nor does it appear on the required/allowed aspect list of the Study Verb, despite it being in both lists.

No Art Needed ("noArtNeeded")

This keeps the game from searching for artwork for this element, and defaults to false. This supposedly offers a small improvement to performance. In general, this should only be true if "isHidden" is true, or if this is an aspect that will only be used as a recipe aspect and XTrigger catalyst. More on XTriggers later.

XTriggers ("xtriggers")

Cards and aspects can both have XTriggers, but the behaviour of xtriggers varies greatly depending on if the element is an aspect or a card. Here we will detail the behaviour of xtriggers for aspects, see further below for cards. Aspects will only proc xtriggers if they have been spawned via mutation. If an aspect is predefined as part of a card, any xtriggers it may have will be ignored.

XTriggers allow a mutated aspect to modify itself. These allow the aspect to transform itself, to mutate the card it belongs to, and/or spawn new cards. Like card XTriggers, aspect XTriggers apply when they are in a recipe with a specific catalyst present.

The “basic” XTrigger is only for transformation: you list the element that catalyzes the trigger, then what this aspect will turn into. This replaces the existing mutation with an equal mutation of a different element.

The advanced use allows for a wider variety of effects. Once again, you start with the element that activates the trigger, but then you open a JSON dictionary. In the list, open an object. This object must have an ID, and may have a ‘morpheffect’, ‘chance’, and/or ‘level’. The ID should be the ID of the new element (spawned, mutated, or transformed). The ‘morpheffect’ should be ‘spawn’, ‘mutate’, or ‘transform’, based on the desired behavior. Chance is the likelihood of the XTrigger’s effects happening; chance defaults to 100%. The behavior of ‘level’ changes based on the morpheffect, but defaults to 1.

  • ‘Spawn’ XTriggers create new cards. When ‘level’ is specified, the card stack that is created will have a size equal to the level. This is equivalent to the ‘spawn’ behavior for card XTriggers.
  • ‘Mutate’ XTriggers apply mutations to the card with the mutated aspect. The amount of the aspect mutated onto the card (which can be negative, though a negative amount of an aspect on a card is considered to be zero) is equal to the ‘level’ of the XTrigger. Unlike the mutations shown in the recipes section below, XTrigger mutations are always additive.
  • ‘Transform’ is the default behaviour, and generally only occurs in the advanced syntax when multiple effects are needed, or a chance or level is desired. The effects of ‘level’ applied to transform-aspect XTriggers is currently unknown, especially when the mutated aspect is already at a non-one level.

Properties specific to Cards

Aspects ("aspects")

Specific to non-aspect elements. A dictionary of aspects contained by the element, where the aspect id is associated to the amount of this aspect. They act as metadata.

Elements as aspects

Very rarely, full card elements can be included in the aspects dictionary, but then that element's sub-aspects are ignored. E.G. adding ‘fragmentknockc’ to a card’s aspect list will not give the card ‘knock’, ‘lore’, or research challenges.

Lifetime ("lifetime")

Specific to non-aspect elements. The card duration. When it reaches 0, it will either transform into the Decay Element, or if there isn't one, it will disappear. If this is set to 0, the element will not have a lifetime, which is the default behaviour if lifetime is omitted.

Behaviour When Purged

When an element is purged through the purge action in a recipe, a “Decayable” element will not be removed like other elements, it will instead decay into its Decay Element. (If the card is in a slot, the card will be fully purged instead. If the card is in a stack, all cards in the stack will be purged at once.)

Resaturate ("resaturate")

Specific to non-aspect elements. Only applies to elements with lifetimes. Normally, as the lifetime reaches 0, the art will desaturate. If this value is true, the card will start desaturated and gain saturation as it reaches 0 (like depleted ability cards, such as Health [Fatigued]).

Slots ("slots")

Specific to non-aspect elements. Elements can have multiple slots. When an element is placed into a verb tile, slots with the appropriate Action ID will appear prior to recipe start. Slots have their own properties (see the Slots section). Element slots are only available when an element is first put into a verb, and thus can only be used at the start of recipe chains. The flexibility of slots allows for craftable recipes with lots of requirements to be met by multiple cards. Element and verb slots often work to control what sorts of cards are able to get into recipe chains to prevent cases where weird elements appear in recipes.

Invocation, Assistant, Offering, and Desire are slots attached to the Sunset Rite element

Hidden ("isHidden")

Specific to non-aspect elements. Cards will always be visible when on the table, but if this is true, they will be invisible while within a recipe.

Unique ("unique")

Specific to non-aspect elements. When true, only one can be present on the board. A new one spawning will cause the old one to fade.

An example of this are rites. If you gain a second of the same rite, the first copy will disappear.

Uniqueness Group ("uniquenessgroup")

Specific to non-aspect elements. A special section that stores an aspect id. Any other elements with the same aspect for their Uniqueness Group cannot exist simultaneously on the board. (Placing one onto the board will cause all others on the board to vanish, just like with the “unique” property.) The uniqueness group is technically an aspect like any other, and can be used for requirements and refinements.

The Aspect Must Exist

It must be stressed that this does not store a simple “uniqueness group name”, but an aspect ID; using a simple string will cause issues. If a string is used instead of an aspect ID, the uniqueness group will work as intended, but any base recipes that contain any element with this undeclared uniqueness group will be unable to spawn additional recipes.

XTriggers ("xtriggers")

Cards and aspects can both have XTriggers, but the behaviour of xtriggers varies greatly depending on if the element is an aspect or a card. Here we will detail the behaviour of xtriggers for cards, see above for aspects. These allow the element to transform itself, mutate itself, and/or spawn new cards when it's used in a recipe with a specific catalyst present. The “basic” XTrigger is only for transformation: you list the element that catalyzes the trigger, then what this element will turn into. Transforming a card with xTriggers keeps any mutated aspects.

The advanced use allows for spawning new cards as well. You once again start with the element that activates the trigger, then open a JSON list. In the list, open a dictionary, which should contain the ID of the new element and a ‘morpheffect’, which can be ‘spawn’, ‘mutate’, or ‘transform’. Advanced XTriggers of any type can be given the ‘chance’ and/or ‘level’ properties. Chance is the likelihood of the XTrigger’s effects happening; chance defaults to 100%. The behavior of ‘level’ changes based on the morpheffect, but defaults to 1.

  • ‘Spawn’ XTriggers create new cards. When ‘level’ is specified, the card stack that is created will have a size equal to the level.
  • ‘Mutate’ XTriggers apply mutations to the card with the XTrigger. The amount of the aspect mutated onto the card (which can be negative, though a negative amount of an aspect on a card is considered to be zero) is equal to the ‘level’ of the XTrigger. Unlike the mutations shown in the recipes section below, XTrigger mutations are always additive.
  • ‘Transform’ is the default behaviour, and generally only occurs in the advanced syntax when multiple effects are needed, or a chance or level is desired. Level is used to determine the size of the new stack.

XTriggers are executed bottom-to-top, so any transformation XTriggers should be declared before any spawn of mutate XTriggers. Otherwise the element will transform, and the other XTriggers won't proc.

Decks

Decks are collections of nouns. A simple, shuffled list, with extras. They are a list of shuffled non-aspect elements (cards) that can be drawn by recipes during their execution. They can be reshuffled on exhaustion, or not.

“Deck” can refer to the deck state (a list of cards in a set order and a list of forbidden cards, which is stored in the game’s save files and does not persist across playthroughs), or the deck object (which is all the data about every instance of a given deck, regardless of what part of which game it may be from… unless the object is changed after the state is made). This distinction is important when decks are modified mid-game: the deck object is changed, but the deck state is not. Cards that might have been removed from the deck object may still be in the deck state, and cards added to the deck object will not be added to the deck state.

Deck states keep track of unique cards (either cards where unique = true, or where ‘uniquenessgroup’ is set) that already exist: when a unique card is generated anywhere in the game, it, and all elements that share a uniqueness group with it, are added to a forbidden list on all decks that contain that element or group. If a card is in a deck state’s forbidden list, it is removed from the deck’s draw list, and will never be shuffled back into the draw list for the rest of the game. This way, duplicate unique cards are not drawn from decks.

Properties of Decks

ID ("id")

Internal name of Deck.  If a deck with this ID already exists, the properties specified will be merged with that of the existing deck. This functionality replaces and streamlines the extends property.

Label ("label")

Displayed name of Deck

Description ("description")

Displayed description

Reset on Exhaustion ("resetonexhaustion")

Whether or not the deck will be reshuffled after all cards have been drawn. Any unique cards in the deck will not be shuffled back in.

Spec ("spec")

The actual list of elements to draw from. There can be duplicates of cards: this is often used to influence the probability of one card to be drawn. This cannot be used to allow a given unique card to be drawn more than once.

Default Card ("defaultcard")

Once the deck has been exhausted and the deck does not reset, this card will be the only card drawn from then on. This is required if your deck doesn’t reset - your mod will not execute otherwise.

Draw Messages ("drawmessages")

A dictionary of element names and messages to be displayed once the cards are revealed depending on which cards are drawn. Used in the Mansus rather than standard deck draws. (Cannot be used outside of Mansus decks.)

Recipes

Description

Recipes are, in essence, sentences. They require nouns and verbs, and transform them in ways that aren’t possible with individual words. Recipes can be chained together to form metaphorical paragraphs, or they can stand on their own. While “Neville”, “Explore”, and “Forgotten Mithraeum” are all terms we can identify and understand, recipes are how we’re able to tie all those things together into their logical conclusion: Neville exploring the forgotten mithraeum.

Order of Effects

Many recipe properties add, remove, or cause changes to elements contained within the recipe at conclusion, or they assess these values to determine what happens next. Recipe execution takes place in a certain order. (This is partially tentative, and may need updates as further tests are done.)

Recipe Start: Slots open, and burn images are displayed.

Alternative recipes are checked, including alt-additional recipes. For each alternative recipe, a random number between 1 and 100 is generated. If that number does not exceed the execution chance of an alt recipe (which defaults to 100), it is considered as a possible alternative recipe during the warmup, or is added to a list of alt-additional recipes to be executed, as appropriate.

During the warmup, recipes whose chances were met will have their requirements reassessed each time a card is placed into or removed from the active slot. This will cause burn images to trigger if the alt recipe has a burn image, and will update the label, description, and loop importance of the Verb if appropriate.

When the warmup timer ends, the active recipe (either a non-additional alt recipe or the base recipe) is “locked in”, and its properties are applied in the following order:

Mutations -> XTriggers from recipe aspects -> XTriggers from elements in the Verb -> deck effects -> effects -> verb halt -> verb delete -> purges -> slot consumptions -> endings.

After that, alt-additional recipes are spawned, but only if their requirements were met at the start of the recipe (when all alts are evaluated) - any element/aspect that appeared during recipe effects will not affect whether alt-additionals fire or not. All expulsions also happen here (as expulsions are part of the firing of alt-additional recipes), so expelled elements are affected by xtriggers/mutations, and any changes in cards can affect what will be expelled. Expelled elements will not be evaluated for links’ requirements. (When deciding which of multiple qualifying elements to expel, the game always chooses the card that has been in the Verb/situation the longest.)

After these, linked recipes are assessed. If none are found, the situation/recipe chain ends, and inductions are triggered, followed by portal (Mansus) effects. Inductions only trigger on face-up cards, so if a card has just been issued via effects or deck effects, it cannot trigger inductions, unless it has lifetime (those are always face-up), or unless it is issued in a recipe prior to the final recipe, then xtriggered, which will cause it to be flipped over.

Properties of Recipes

ID ("id")

This is the internal identifier for the recipe, you will always use this to reference the recipe. If a recipe with this ID already exists, the properties specified will be merged with that of the existing recipe. This functionality replaces and streamlines the extends property.

Label ("label")

This is the name of the recipe that is displayed to the player

Max Executions ("maxexecutions")

This is the maximum number of times you want to allow a recipe to be executed. Every recipe is tracked whether it has this property or not, but this is the only evaluation that can be made against that value.

Warmup ("warmup")

This is the length of the timer that counts down after starting a recipe. If the recipe is executed as an Alternative Recipe, this property is not needed, and will be ignored.

Craftable ("craftable")

This boolean property enables or disables the start button when recipe conditions are fulfilled. Recipes can always be executed by other recipes; this just controls which recipes players can manually start through the verb tiles. It defaults to false if not stated.

Hint Only ("hintonly")

This disables the start button but still displays the recipe when true. If the requirements for a hint recipe and a normal, craftable recipe are met, the normal recipe is preferred. Hintonly recipes do not need to also have craftable set to true.

Signal Important Loop ("signalimportantloop")

This changes the color of the recipe warmup timer to a yellow-gold color and plays a sound that lets you know something important is happening. Usually true for ending recipes. Is also true for the recipe that plays when the season of sickness recipe fails to grab a health card or follower.

Internal Deck ("internaldeck")

This is a deck of elements that is only accessible to the recipe accessing it. When this has a value, it will be drawn from exactly as it is told to in the internal deck specifications. It is preferable to use external decks, since these can be modified by other mods, while internal decks can only be overwritten.

Slots ("slots")

This is basically the slot for a Quick Time Event that is available when the recipe is warming up, usually used to give the players input on a decision for the recipe. Such as the decision for a rebellious spirit, either supplying a reason or passion or choosing to ignore it completely. Recipes can have only one slot, see the Slots section for more information.

Start Description ("startdescription")

This is the description displayed before the recipe is started and while the warmup counter executes.

Description ("description")

This is the description displayed after the recipe has been completed. This text is never displayed if the recipe routes to another linked recipe, as the action is not considered completed until a recipe ends without starting another one.

Requirements ("requirements")

This is an inclusive set of conditions that is evaluated whenever the recipe is encountered in a recipe link to decide whether or not to execute the recipe. The list of elements (and aspects) must all be present within the recipe’s element stack. This can also be used comparatively, having a requirement of "element_1" : "element_2" requires that you have a greater than or equal quantity of "element_1" as "element_2". Furthermore, negative numbers are allowed, and function as "less than", allowing -1 to represent 0, -2 to represent having 1 or 0, etc.

Table Requirements ("tablereqs")

This is an inclusive set of conditions that is evaluated whenever the recipe is encountered in a recipe link to decide whether or not to execute the recipe. The list of elements (and aspects) must all be present on the game board, and not in a verb tile. Accordingly, elements that are in slots, both for ongoing and yet unstarted recipes, and elements that aren’t yet picked from the verb after the situation ended, are considered to be outside of the tablereqs scope. Like requirements, tablereqs support less than and comparisons.

Extant Requirements ("extantreqs")

This is an inclusive set of conditions that is evaluated whenever the recipe is encountered in a recipe link to decide whether or not to execute the recipe. The list of elements (and aspects) must all be present somewhere within the game. They can be in another recipe, in the player’s hand, or on the board. Like requirements, extantreqs support less than and comparisons.

Effects ("effects")

This is a set of operations to apply to the element stack. When an element is added, it will be added as a card of that element type. When an element is removed, the first encountered source of that element will be removed. This could be a card of that element or it could be a card with that element as an aspect. Negative values can be used to remove elements. If an aspect is specified for removal, an entire card with that aspect will be removed. Comparisons are also available for effects; for example, having "element_1" : "element_2" will create as many new "element_1"s as there are "element_2"s in the recipe stack.

Purge ("purge")

This is a set of operations to apply to the game board. It will decay up to the specified amount of the element sources. If a specific element type is specified, it will decay to whatever is indicated in that element's "decayTo" property, even if the element has no lifetime to cause it to decay naturally. If no element is specified under "decayTo", the element will be destroyed. If Purge is used on an aspect, all elements with the aspect will normally be destroyed since aspects do not normally have a decayTo property. However, if for some reason the aspect does have a decayTo property, then instead of destroying elements with the aspect, elements with the aspect will be decayed. This works well because the element that the aspect has been set to decayTo will be ignored; instead, each element that has the aspect will decay according to its own decayTo aspect. If the individual element has no decayTo aspect, then it is destroyed.

Aspects ("aspects")

This is a set of operations to apply to the recipe’s aspect stack. The aspect stack is the sum of all the aspects of the cards in the element stack. The effects of this property do not apply to the cards themselves, just to the stack produced by their sum. This is usually used to introduce temporary aspects for the purpose of triggering an XTrigger reaction, since the aspects disappear after the recipe ends, and don't proc inductions.

Deck Effects ("deckeffects")

This is a set of deck IDs that are drawn from the specified number of times and added to the element list at the conclusion of the recipe.

Halt Verb

This halts the specified verb up to the given (and seemingly required) number of different tokens. The active recipe transitions into a closable form that is signified by the clickable button at the bottom left corner of the token. Ending the recipe in the verb will delete the verb token and return any cards still in the stack.

Delete Verb

This halts the specified verb and deletes the token at the same time, also deleting any elements within the verb at the time of deletion.

Action ID ("actionId")

This is the Situation Token ID that is used to execute the recipe, if it corresponds to a defined verb, the verb will persist once the recipe ends. If the token for a recipe is already present on the board, it cannot be spawned again. When calling an alt-additional recipe, the actionId of the new recipe should be different from that of the original recipe. This needs to remain consistent throughout a recipe chain, as saving and loading uses an active recipe's actionId to determine which tile to place on the board.

Ending ("ending")

This is the ID of an ending you’d like to trigger at the conclusion of this recipe.

Signal Ending Flavour ("signalEndingFlavour")

This will change the colour of the warmup circle around the tile, and trigger fancy music. If it is not one of the predefined and hardcoded values it will default to None. The predefined values include Grand for a golden effect, Melancholy for a red effect, Pale for a white effect, and Vile for an edge-green effect.

Burn Image ("burnimage")

This is the image filename of a png file located in the “images/burns” folder that you’d like to display on the board when the recipe begins.

Portal Effect ("portaleffect")

This is the property that tells a recipe to transition into the Mansus and which door to open to. This causes the recipe to draw a card from each deck associated with the door and let you choose from them on the board.

Mutations ("mutations")

Mutations are used to add or remove aspects from an element. Mutated aspects remain even when the element changes via xtrigger or decay.  There are a few properties needed to set a mutation.

Filter ("filter") - used to find the element that will be modified. Filters can be elements or aspects.

Mutation Aspect ("mutate") - the aspect we are adding/removing on the element.

Level ("level") - a positive or negative value to represent how much of the given aspect we are adding or taking away.

Additive ("additive") - a boolean value that decides whether this change replaces the stated value with the current one, or adds/subtracts the value from the base element's values. This defaults to false.

Alternative Recipes ("alt")

This is a list of Recipe Link specifications. Every time the aspect stack of the recipe changes, this property is evaluated for a valid recipe and then the active recipe is switched to it, preserving the timer but only executing the effects of the specified alternative recipe. Recipes are evaluated in the order of the list; evaluation stops once a new recipe is found, with the exception of additional recipes, which are always evaluated.

ID ("id") - This is the ID of the recipe this link points to

Chance ("chance") - This is a value from 0-100 that represents the percent chance this recipe will be selected from among the list of recipes when the conditions are met. If the requirements for the new are not met, the recipe will not run, regardless of the chance.

Additional ("additional") - This is whether or not a new situation token will be generated for the specified recipe. If the Additional property is true and an alternative recipe is chosen for execution (requirements are met and it passes the chance roll) then it is added to a queue that is iterated through upon completion to create situation tokens. See Order of Operations for more details.

Challenges ("challenges") - This overrides the chance property with an aspect-based challenge. There are two types of challenge, base and advanced. A base challenge scales the chance into a histogram based on the level of the specified aspect in the current recipe’s aspect stack. For Base, the chances are: 0 for 0%, 1-4 for 30%, 5-9 for 70% and 10+ for 90%. For Advanced, and then assign the chance based on the following: 0-5 for 0%, 6-10 for 10%, 11-15 for 30% and 16-20 for 70%, and 20+ for 90%.

Expulsions ("expulsion") - This is a dictionary of filters and limits. An expulsion will try to grab up to the specified number of the source of the specified element in the filter. It will grab either cards or cards with the element as an aspect. Once it has grabbed all the cards that it is told you, it will carry them into a newly generated situation token if Additional is true. Nothing happens if Expulsions are set while Additional is false, the cards are basically put back.

Linked Recipes ("linked")

A list of recipes to be executed after the current recipe is resolved. This is different from alternative recipes because it does not replace the current recipe's effects with the new one. Also, additional recipes must be alternatives, not linked. A few properties are required.

ID ("id") - the ID of the recipe to be executed.

Chance ("chance") - a number from 0 to 100, inclusive. Represents the percent chance of the recipe executing.

Challenges ("challenges") - a way of determining chance based on how much of a given aspect is currently in the recipe. First you declare an aspect, then you declare the challenge to be a "base" or "advanced" challenge, which represent different statistical chances.

"base" challenges have 0% chance for 0 aspect, 30% chance for 1, 70% chance for 5, and 90% chance for 10.

"advanced" challenges have 0% chance for 0 aspect, 10% chance for 5, 30% chance for 10, 70% chance for 15, 90% chance for 20.

If challenges and chances are used, the challenges overwrite the chance property. 

Slots

Description

Slots are...slots, defined by elements, verbs or recipes. They act as filters, giving the possibility to put cards into them, if they match their requirements.

Properties of Slots

ID ("id")

slots all have an ID; it's only used by the parser.

Label ("label")

the text displayed above the slot.

Description ("description")

Text displayed in the top right when the slot is clicked on.

Required ("required")

A list of required elements/aspects. Any card that meets any one of the properties in Required can be put in the slot, unlike the requirements property for recipes. Only uses positive values.

Forbidden ("forbidden")

A list of forbidden elements/aspects. Any card that meets any one of the properties in Forbidden cannot be put in the slot, even if it qualifies for the Required list.

Consumes ("consumes")

After the recipe concludes, any element in this slot will be destroyed. Shows the little candle at the bottom of the slot.

Action ID ("actionId")

Specific to element slots. The verb name where the slot will appear when the element is inserted.

Greedy ("greedy")

Specific to recipe slots. When this is true, the slot will pull a qualifying card off the table into the slot automatically, rather than letting the player choose whether to insert a card, or which card to insert.

Verbs

Description

Verbs are verbs. They are the things that you do. Things that you initiate. If a verb is defined in a file, it will stay on the board after it is called once. These are Permanent Verbs. Undefined Verbs, or Temporary Verbs (sometimes called tokens), will disappear once their recipes resolve and the situation ends.

Properties of Verbs

ID ("id")

The internal name of the verb. Determines the image used, found in images/verbs. Permanent and Temporary verbs check for images here. The actionId of recipes and elements should match this value if you want the action token to persist.

Label ("label")

The name of the verb displayed to the player when clicking on the token while no recipe is running.

Description ("description")

Text displayed in-game when clicking on the token while no recipe is running.

Slot ("slot")

Verbs can only have one slot. See slots for more information.

Tools and Resources

Official Discord

For modders coming from another source, the Weather Factory Discord Server has a #modding channel where modders gather and help each other (they’re the main source of contributors for this guide).

Frangiclave

Frangiclave (frangiclave.net) is an online reference of the game files, parsed and made easier to read. It is the best reference you can find if you cannot read the game files at this moment, or prefer a more readable way of reading the game files.

Carcass Spark

Carcass Spark (https://github.com/justastranger/CarcassSpark/releases) is a visual tool to read the base content, create mods, and work on them. It is slower than writing them by hand, is easier to use, and prevents a lot of typos. You can also do a lot of things with it, like checking where a recipe/element is used, or directly see and edit the objects as JSON, which is a great way to check the syntax of something.

Warning

The Carcass Spark tool is being actively developed, and thus is prone to bugs that can result in content loss. The given link is often a few weeks behind the latest version (containing all the hotfixes). You can always find the latest version of the tool as a pinned message in the #modding channel of the Weather Factory Discord Server (see above).

The Steam Workshop

The Steam Workshop (called the “Invisible Serapeum”) is the official place where the mods are published.

Achieve Integrations

Achieve Integrations is a coremod with a number of compatibility fixes and modding tools, including the following:

  • Slot overrides on verbs, as well as some elements and recipes, to add generic allowed/forbidden aspects, enabling many mods to create elements for these slots without having to change the slots themselves.
  • Changes to jobs and rites to finer allow control over what's allowed in them and when they can be executed.
  • New aspects that exist to describe cults and rites, overhauling the recruit/promote/exalt processes to allow for new, custom cults.
  • A new developer-only Verb and recipe (modded_xtrigger) that can be used to easily test XTriggers and inductions.

Information that follows is written to describe AchInt 2.5.0. Information here is likely to remain true, but may be made incomplete by future updates.

Verbs

        To allow an element to be placed in the main slot of a Verb, give it the aspect “modded_(Verb name)_allowed”, replacing (Verb name) with the name of the Verb. (In Exile, the name of the Verb is the name in the code, not the name in-game: relinquish, scout, send, travel, and use.)

To prevent an element from being placed in the main slot of a Verb, give it the aspect “modded_(Verb name)_forbidden”. (Be aware that giving the same card both the forbidden and allowed aspects of a Verb will cause the card to be forbidden.)

It is recommended that any mods adding their own Verbs go ahead and add aspects that match this pattern for their own Verbs, and configure those Verbs to allow/reject cards with those aspects accordingly.

Rites and Rituals

New aspects that describe rites:

  • modded_rite_accepts_promise: This is a rite that will accept a promise card instead of a desire card. In AchInt, this aspect is only given to the Rite Intercalate. If you create a new rite that accepts a promise, you are encouraged to give it this aspect.
  • modded_rite_consumes_(resource): This is a rite that has a consuming slot for a given resource. Resources include follower, influence, ingredient, lore, and tool. For example, Rite of Rebel Striving (ritefollowerconsumetool) is given the modded_rite_consumes_tool aspect. When you create a rite with a consuming slot, or give a new consuming slot to a rite, you are encouraged to give the appropriate rite_consumes aspect to that rite.

New aspects that regulate rites:

  • modded_rite_consumes_forbidden: An element with this aspect cannot be put into a consuming slot in a rite. For example, giving it to a Lore card will prevent that card from being used in Rite of the Watchman’s Sorrow, or Rite Intercalate.
  • modded_rite_consumes_required: An element with this aspect cannot be put into a non-consuming slot in a rite. For example, giving it to a Lore card will prevent that card from being used in any rite other than Rite of the Watchman’s Sorrow or Rite Intercalate.
  • modded_rite_desire_(allowed/forbidden): An element with this aspect (can/can’t) be placed in the desire slot of any rite.
  • modded_rite_(resource)_(allowed/forbidden): Elements with the appropriate aspect for a given resource type (follower, influence, ingredient, lore, tool) can/can’t be placed into a rite slot that accepts that resource type. For example, a tool with modded_rite_tool_forbidden cannot be placed in the Tool slot of Rite Intercalate, Rite of the Watchman’s Sorrow, Rite of Rebel Striving, etc. Meanwhile, a Tool with modded_rite_influence_allowed can be placed into the Influence slot of any rite that accepts an influence.
  • modded_rite_intercalate_forbidden: Elements with this aspect cannot be used in any slot of Rite Intercalate.
  • modded_rite_intercalate_required: Elements with this aspect cannot be used in any rite other than Rite Intercalate.

New aspects that regulate rituals:

  • modded_normal_rituals_forbidden: An element with this aspect cannot be used in any vanilla ritual that doesn’t involve a desire or promise. In other words, this aspect prevents summoning any vanilla spirit, creating a Risen, and enacting Forge’s Redemption, The End is Beautiful, Passion’s Gyre, and Reason’s Glory, as well as hints that relate to any of those.

Cults

        New aspects that describe cults (and enforce those descriptions!):

  • cult_exalts_(principle): This aspect is given to cults capable of creating Exalts of the given Principle using the normal exaltation recipes. By giving a cult this aspect, you give it the ability to create Exalts of that type.
  • cult_exalts_none: This aspect prevents cults from forming Exalts of any kind through the normal means. In addition, this aspect implies that the cult is incapable of forming Exalts through any means, though this trait is not enforced. If you want your cult to produce Exalts through other methods, it’s suggested that you give the cult the modded_cult_exalt_disciple_forbidden aspect instead (see next).
  • modded_cult_exalt_disciple_forbidden: Elements with this aspect cannot be used in the normal process of promoting a disciple to exalt. This aspect can be applied to the cult, the associate, the ceremony, or the trappings, and the recipe will not engage.
  • modded_cult_promote_believer_forbidden: Elements with this aspect cannot be used in the process of promoting a Believer to Disciple, akin to the way that the above aspect works with exalting Disciples.
  • modded_cult_recruit_acquaintance_forbidden: Elements with this aspect cannot be used in the process of recruiting an acquaintance into a cult through the normal means.
  • modded_cult_recruit_independent_forbidden: Elements with this aspect cannot be used in the process of recruiting an independent (Maverick) into a cult through the normal means.

Online Translation Tool

CS Online Translation Tool (https://genroa.github.io/Cultist-Simulator-Translation-Tool/) is a web app (that can be installed to be used offline) made to help translating CS content files. It is limited and can be laggy if the entire set of base game recipes is loaded, but “it works”.

Only Accepts Valid JSON

Some game content files have JSON mistakes that most JSON parsers cannot read. You can fix them yourself, or use Carcass Spark to export a cleaned version of them (see the Tools menu of the software).

How Stuff works

Followers

Required Aspects - followers need a uniqueness group. An aspect for uniqueness should be created. All elements of the follower should have their uniqueness group set to this aspect.

Additionally, you should create a hidden aspect for all of your followers that identifies them as yours, particularly if you want to make them romanceable. This aspect is used for recipes later to let your follower mod work with other follower mods. Give this aspect to all of your follower's elements. If you're adding more than one follower, use one aspect for all of them.

Required Cards - Followers have a few different required elements that have to be configured in a certain way. All elements of all followers should be mortal. All elements of all followers should have an xtrigger on killmortal to become a corpse, and an xtrigger on derangemortal to become a lunatic.

  • Acquaintance - should have the acquaintance aspect. Should have xtrigger on recruiting to become a believer, and an xtrigger on rebellion to become a renegade. This is the card that is put in the acquaintances deck. If you'd like your follower to be recruitable the normal way through the talk verb, add them to this deck.

  • Believer - should have the follower aspect. Believers conventionally have 2 of the aspect of their associated Principle. Romanceable followers have the desired lust aspect. Believers should have an xtrigger on promotiontodisciple to become their disciple element, an xtrigger on capturefollower to become their prisoner element, and should also have an xtrigger on rebellion to become their rival form.

  • Disciple - should have the follower and disciple aspects, and conventionally have 5 of their associated Principle. If they have a secondary principle, it is conventionally set to 4. Romanceable followers should have the same lust aspect from their believer element. If your follower has an exalted element, the disciple should have the potential aspect, and an xtrigger for promoting: this xtrigger varies depending on their main Principle. Aside from exalting, all disciples should have an xtrigger on capturefollower to become prisoners, and an xtrigger on rebellion to become a stage 2 rival.

  • Exalted - should have the follower, disciple, and exalted aspects. Conventionally, exalts have 10 of their main Principle, and 7 of their secondary if applicable. If romanceable, exalts should have the same lust aspect as they did as followers/disciples. Exalts have an xtrigger on rebellion to become Long in the Making.

  • Prisoner - prisoners all have the prisoner aspect, and conventionally have three grail, three lantern, and three heart. For follower prisoners, they have five of their main Principle as well. If their main principle is Grail, Lantern, or Heart, the values are not added together, and are instead set to 5. Prisoners conventionally have a lifetime of 600, and decay to corpses.

  • Rival - Rivals have 2 rival marks, the independent aspect, and the suspicious aspect. They have xtriggers on rival_promotion to become a Stage 2 rival, and on recruiting to become believers.

  • Stage 2 rival - A promoted rival has 3 rival marks, as well as the independent, suspicious, and disciple aspects. Xtriggers on recruiting to become disciples, and on rival_promotion to become Long in the Making.

  • Long in the Making - final form for rivals. They have 4 rival marks, and the independent, suspicious, and exalted aspects. Xtrigger on recruiting to become Exalt.

Required Recipes - Only romanceable followers need recipes. If your follower is not romanceable, you're done! Otherwise, you need to create some recipes, and modify some base game ones.

The base game recipes use description text with a refinement to choose text based on which follower you're speaking with. This text cannot be modified without replacing it entirely. If you replace it entirely, no other follower mods can be used at the same time as yours. So instead of overwriting the base game recipe descriptions, it is better to add equivalent recipes with your own descriptions.

Starting romance recipes

First are the romance recipes that occur from talktofollower to begin a romance. First, we create a lust recipe recipe that is exclusively for your followers. You will need a different recipe for every lust, and each lust has an associated location. Regardless, your recipe will look like the associated A_talktofollower_… recipe, except the ID must be different, and you will add a requirement for your oc aspect created at the start. Then you will modify the description text to have a section for each of your followers with this particular lust. (Note: if you have more than one follower, use the selective text syntax based on your follower's uniqueness aspect. This way it won't matter what promotion level your follower is at.) Just like the base recipe, you should link to L_talk_roseopportunity. Once the recipe is made, prepend it to talktofollower.

This next step isn't strictly necessary, but is good to do. You should modify the base game A_talktofollower_… recipe's requirements, adding the requirement to NOT having your oc aspect. Just in case.

Now we have to do a similar process, but for the applicable A_startrose_..._initiate recipes. Again, you'll make an identical recipe with a different ID, make it require your oc aspect, and give it different descriptive text. This recipe should then be prepended to alt for L_talk_roseopportunity. Then, once again, go add to the requirements for the base game A_startrose_..._initiate recipe to require not having your oc aspect.

Next, we need to add descriptive text for the season of ardours. We'll follow the same process of copying a base game recipe, but its easier since we only have to do it once for the last recipe. Find the applicable L_continuerose_..._success recipe. Make a recipe just like it, but with a different ID, a requirement for your oc aspect, and a description for your applicable followers. Prepend this recipe to the associated L_continuerose_… recipe. Then add to the requirements of the base game L_continuerose_..._success recipe to require not having your custom oc aspect.

Optionally, we can add our followers name to the romance ending recipes. The base game will automatically use "My lover" if none of the base game followers are in the recipe. If this is satisfactory, skip this section. If not, we use the previous method again.

First for the A_roseend_expulsion recipe. Make another recipe like it, with a different ID, modified flavour text, and a requirement for your custom oc aspect. This recipe must get prepended to the alternative recipes of L_activatejealousy with additional set to true, and an expulsion on follower with a limit of one. Once this is done, the base game L_roseend_expulsion recipe must be modified to add the requirement of not having your custom oc aspect. This step isn't optional here because we're dealing with additional recipes, and we don't want both executing.

Now we do this again for the L_roseend recipe. Change the ID, add a requirement for your custom oc aspect, and change the description text. Then prepend your recipe to linked for L_continuerose_chance, L_continuerose_enlightenment, L_continuerose_power, and L_continuerose_sensation. Optionally, modify the base game L_roseend recipe and add a requirement to NOT have your custom oc aspect.

Lastly, we just need to add an Ever After victory. First, we will set up the recipe that lets you achieve this victory. The base game has recipes for this called [name]victory. Make a recipe based off these ones, but with the requirement for your follower's uniqueness aspect. This recipe should then be prepended to belovedfarewell AND to belovedfarewelldream. Do this for each of your followers.

Now you just need to create an ending for your followers. Conventionally, Ever After endings use the Melancholy ending flavour and the DramaticLight animation.

Romance Categories

NOTE: While adding new romance options is possible, it is not currently possible to remove romance aspects from followers. New ones can still be added, but giving a follower two romance categories could cause minor issues.

Adding new romance Categories is easier than giving romance options to followers.

First, you need a hidden aspect for your romance type. This aspect should then be given to all applicable followers.

Next, you need to add a recipe to talktofollower. This recipe will functionally be identical to the other romance recipes, A_talktofollower_[location]_lust[type]. The start description should have refinement text with a description for each follower using this romance type. It should require your new romance aspect, and needs to require something else as well. Conventionally this is a location, it could be something else as well provided it doesn't interfere with other talktofollower recipes. This new recipe should be prepended as an alt recipe under talktofollower, and should link to L_talk_roseopportunity.

Our next recipe will be functionally equivalent to the base game A_startrose_[type]_initiate recipes. Ours should require a specific temptation. This should be done with an aspect that is present on all forms of the given temptation (ie A_startrose_power_initiate requires at least 1 powermark, this way all levels of ascensionpower will qualify). It should have refinement text in the description for all followers with this romance option. In addition to giving the romanticinterest aspect via mutation, this recipe conventionally gives an additional point in an associated principle via mutation as well.

Now that followers can start romance through the new option, we need to add a couple recipes to the season of ardours.

We will make a new recipe functionally identical to the L_continuerose_[type] recipes. Ours will require our new romance aspect. This new recipe will have a slot that needs to be filled. Funds and dread are used to end your romance, but anything besides those could be used to satisfy it. This recipe will have it's last alt recipe be the recipe we're creating next. This recipe should be prepended to the linked recipes in roserouter.

Our next recipe is a custom version of L_continuerose_[type]_success, and should require whatever we previously decided would be needed to satisfy the romance. Conventionally it gives contentment, and exhausts its required skill card. If using something different, you might consider exhausting or removing it. It has refinement text in the description for every associated follower.

Expeditions: Vaults & Obstacles

Description

Vaults are places you can explore to do Expeditions. Expeditions are a gameplay system started when exploring with a vault card. Its main loop is :

  • Vault specific start description
  • Expedition loop
  • Send Funds or Followers
  • The next obstacle to encounter is selected (alt recipe). You can add Funds or Followers during this phase.
  • The obstacle is resolved (in the base-game, it is an aspect challenge)
  • If there are no more obstacles, display the vault specific success description and draw rewards somehow (usually from a dedicated deck)
  • If you don’t have followers left or no money anymore, the Expedition ends.

The Expedition System is structured this way:

Blue: Recipes, Green: Alternative Recipes. Recipes with * are additional. Alternative recipes are stacked under their base recipe.

How to create new Vaults

Creating new vaults using existing obstacles is fairly easy.

Custom Obstacles

Custom Obstacles are explained down below. This section only deals with the creation of new vaults.

New vaults only need theses :

  • A new element, vault[tier][name], with aspects {"vault": 1, "location": 1, vault[tier]: 1}. It should have the same two slots as other vaults, ideally, but you can do whatever you want.
  • A new recipe, vault[tier][name]_setup, with explorevault_expeditionbegun as a linked recipe.
  • It must have {"funds": 1, "follower": 1, [new vault element]: 1} as a requirement.
  • It must be craftable.
  • It should add in its effects the elements of the obstacles that will be encountered.
  • A new recipe, vault[tier][name]_success. It should remove the vault card from the verb, and spawn the "rewards". The base-game draws rewards from dedicated internal decks for each vault. (AchieveIntegrations changes that to external decks)
  • Appending vault[tier][name]_success to explorevaultsuccess' linked recipes.

How to create new Obstacles

To create new obstacles, you need:

  • A new element, [obstaclename], describing the obstacle itself. Any vault using it will have to spawn it in its setup recipe's effects
  • At least one new recipe, [obstaclename]_encounter. It should have {[obstaclename]: 1} in its requirements. Technically speaking, past this entry recipe, the obstacle logic could be anything, the expedition system doesn't care. It only needs to eventually go back to the tick recipe, and usually remove the [obstaclename] card from the verb on success. The way base-game obstacles work, is:
  • By using the encounter recipe and defining several alternative recipes, all displaying the same slot (assistance, same slot as the tick recipe).
  • The alternative recipes are used when enough of some aspect is present in the verb
  • The obstacle defines failure and success recipes. Failure recipes for guardians and perils have a 70% chance to call an alternative additional recipe, explorevault_casualty, where they expel a follower. Curses have a 100% chance on failure to spawn a curse verb. They route back to the tick recipe. Success recipes remove the obstacle from the verb and then route to the tick recipe. Failure recipes leave the element to be faced again, with the exception of curses, which are removed on failure as well.
  • The encounter recipe always links to the failure recipe
  • The alternative recipes link to the success recipes, with a probability that depends on what amount of aspect they represent (usually, 30%, 70% and 90%), and a link to the failure recipe
  • Append or prepend the encounter recipe to the explorevaulttick linked recipes list.

Obstacles Order

Because obstacles are met in the order of the explorvaulttick linked recipes list, and new obstacles can only be appended or prepended, you cannot really choose the order in which obstacles are met. The order for base-game obstacles is: treachous ground, forest, mountains, desert, sea, watchers, monks, serpent, dead, soucouyants, long, hiddendoor, wardeddoor, dread, visions, affliction, nowhere. Some mods (like AchieveIntegrations) try to improve this by overhauling the expedition system to add "phases" you can select for your obstacles, giving more flexibility.

  • In the explorevaultsuccesslink recipe, either add {[obstaclename]: -1} to the requirements, or give your obstacle one of the base-game obstacle categories (peril, guardian, seal, or curse), since the base-game groups are what the successlink recipe checks for.
  • In the explorevaultfailure_nofunds and explorevaultfailure_nofollowers recipes, add {[obstaclename]: -10} (or any value deemed big enough to remove any instance of these obstacles from the verb) to the effects. The base-game grouped obstacles by type, giving them an aspect, and uses it in these effects.
